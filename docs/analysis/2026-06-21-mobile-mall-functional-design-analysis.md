# 2026-06-21 移动商城开源项目功能设计深度分析与 litemall 增强建议

## 目的

对三个开源移动商城项目进行**纯业务功能设计层面**的深度分析，提炼可借鉴的功能模式与差异化设计思想，并据此为 `litemall` 后续的「移动商城功能增加」与「后台设计完善」提供结构化的功能增强建议清单。

本报告不是技术选型或架构分析，不评价技术栈、代码结构或框架实现。

## 分析对象与口径

### 三个开源项目

| 项目 | 路径 | 技术栈 | 产品定位 |
| ---- | ---- | ------ | -------- |
| 芋道商城移动端 | `/Volumes/data/sources/yudao-mall-uniapp` | UniApp + Vue3 | 全端、全玩法、SaaS 多租户的生产级电商移动端 |
| c-shopping-rn | `/Volumes/data/sources/c-shopping-rn` | React Native + Expo | "小而美"的中文内容型电商 App MVP |
| 新蜂商城 Vue3 | `/Volumes/data/sources/newbee-mall-vue3-app` | Vue3 + Vant | 教学骨架级 B2C 移动 H5 |

### 增强目标基线

- `litemall`（`/Volumes/data/sources/litemall`）：单店、单支付通道（微信）、无分销、无 IM、无装修的"小型 B2C 自营商城"参考实现。已有完整的交易主链路，但深度运营能力薄、营销玩法少、报表基础。

### 口径约定

- 本报告聚焦**业务功能设计**（用户角色、功能模块、业务流程、业务规则、状态机、玩法机制、运营能力）。
- 标注「前台」=终端用户可见可操作；「后台支撑」=需要后台配置/运营才能成立；「占位」=界面存在但功能未实现。
- 三项目分别简称 **芋道**、**c-shopping**、**新蜂**。
- 增强建议按 P0/P1/P2 标注优先级（P0=主链路补全/合规/高 ROI；P1=运营核心能力；P2=体验增强/差异化）。

## 第一部分：三项目功能设计分项深度分析

### 1. 用户与会员体系

#### 1.1 芋道（最完整）

- **5 种登录方式**：账号密码、手机号+短信验证码、微信公众号网页授权、微信小程序一键登录、三方社交登录（预留钉钉/抖音）。
- **双令牌管理**：access_token + refresh_token 自动续期。
- **登录后编排**：刷新用户信息 / 钱包 / 订单数 / 优惠券数 / 购物车、配置全局分享、绑定推广员、必要时弹"绑定手机号"弱提醒。
- **个人资料**：头像、昵称、性别、手机号绑定（独立校验，支持短信改手机/小程序快捷获取）、登录密码（设置/修改/重置）、三方账号绑定与解绑。
- **会员等级**：后端有完整等级体系，订单页可见会员价/会员折扣（`vipPrice`），签到获得经验值联动等级，但前端无独立"等级中心"页。
- **积分**：独立"我的积分"页（余额、收支 Tab、时间段筛选、积分流水），积分用途=下单抵扣 / 积分商城兑换 / 签到获得。
- **签到**：后台可配置连续签到奖励规则（第 N 天奖励 X 积分）、连续/累计天数、已签置灰、签到结果弹窗（积分+经验+连续天数）。
- **用户标签**：后端支持，前端未独立展示。

#### 1.2 c-shopping（最简但本土化彻底）

- **注册/登录**：邮箱+密码（注册即登录），无三方登录，无游客注册。
- **半游客模式**：浏览免登录；加购、查看购物车、结算、写评价、查看订单/收藏/地址/足迹/账户信息时强制登录（`mustAuthAction` 拦截）。
- **Token**：登录返回 token，前端 header 携带 + AsyncStorage 持久化 + 启动自动恢复。
- **个人资料**：**仅两项可编辑**——姓名（≥3 字符）、电话（11 位数字）。
- **会员等级 / 积分 / 成长 / 签到 / 偏好**：**全部无**（评价引导文案提到"赠积分"但无积分 UI）。

#### 1.3 新蜂（教学最小集）

- **注册/登录**：账号密码（注册/登录同页切换），**图形验证码本地生成+本地校验**（Canvas）。
- **个人资料**：昵称、登录名、个性签名、登录密码；**头像固定占位**，无性别/生日/手机号。
- **登出/修改密码**：支持；**找回密码**：无（只能登录后改）。
- **Token 机制**：服务端 `resultCode=416` 表示未登录/失效，前端统一跳登录页。
- **匿名浏览 + 登录拦截**：流量页匿名可访问，转化页强制登录。
- **会员等级 / 积分 / 成长 / 签到**：**全部无**。

#### 1.4 小结与设计模式提炼

| 维度 | 芋道 | c-shopping | 新蜂 |
| ---- | ---- | ---- | ---- |
| 登录方式数 | 5 | 1（邮箱密码） | 1（账号密码+图形码） |
| 三方登录 | 微信生态全覆盖 + 社交预留 | 无 | 无 |
| 会员等级 | 有（后端完整，前端展示弱） | 无 | 无 |
| 积分体系 | 有（账户+流水+抵扣+兑换） | 仅文案钩子，无 UI | 无 |
| 签到 | 有（可配规则） | 无 | 无 |
| 半游客模式 | 部分 | 有（最清晰） | 有 |

**可提炼的设计模式**：
1. **半游客模式 + 关键动作拦截**（c-shopping 最清晰）：浏览免登录降低流失，加购/下单拦截提升注册转化——这是移动电商的转化率黄金法则。
2. **登录后编排流**（芋道）：登录不只是鉴权，而是一次"状态同步"（刷新资产、绑定推广关系、触发弱提醒）。
3. **图形验证码本地化**（新蜂）：零成本防机器人，适合教学/小型项目。
4. **会员体系分层**：等级（身份）、积分（流通货币）、成长值（升级进度）、签到（日活抓手）是四个正交概念，芋道做了完整闭环。

### 2. 商品与目录

#### 2.1 芋道

- **多级树形分类**（至少两级，可扩展三级），3 种 UI 样式切换（一级大图/一级瀑布/二级左右），首页分类 tab 全局状态传参定位。
- **商品详情承载元素最丰富**：轮播、限时折扣/会员价横幅、价格+划线原价、销量、优惠券标签、满减送标签、领券入口、标题+副标题、SKU 规格弹窗、评价卡片（近 3 条+总数）、详情富文本、**活动跳转提示（聚合拼团/秒杀/砍价）**、底部操作栏、分享海报。
- **SKU**：多规格（propertyName+valueName），加车前强制选规格。
- **搜索**：本地历史（最多 10 条，置顶去重，可清空，localStorage 持久化）。
- **筛选排序**：综合（综合/价格升/降二级菜单）、销量、新品优先；单列大图/双列瀑布布局切换。
- **商品推荐**：靠 DIY 装修组件（s-goods-card/shelves/scroll/column/item）配置商品 ID 列表或分类筛选，**无算法推荐**。
- **亮点**：**结算信息实时拼接到列表卡片**（`getSettlementProduct`），把营销价、优惠券、满减送到列表卡片上展示。

#### 2.2 c-shopping

- **三级分类树**：左侧一级 rail（圆形图标+文字）+ 右侧二/三级，商品列表页顶部同级子分类横滑。
- **一级分类带主题色**（`colors.start`）：进入分类整体色系变化。
- **商品详情**：多图轮播、标题、颜色/尺寸选择、缺货提示、属性 Info 列表、免费送货卡片、介绍（>300 字折叠）、相似商品横滑、规格表（>7 条折叠）、评价区、底部智能操作条（未加购显按钮，已加购显数量加减）。
- **SKU 漏洞**：颜色+尺寸同时存在时，购物车合并判定只按一维（color.id），复杂 SKU 场景会误并单。
- **搜索**：1.2s debounce，分页，无历史/热搜/联想/筛选叠加。
- **筛选**：仅库存/特价开关 + 价格区间。
- **排序**：4 种（最新/最畅销/最便宜/最重磅）。
- **首页 7 模块 feed 节奏设计**（亮点）：轮播→圆形分类→折扣横滑→2 列专题网格→**编号畅销榜**→大图专题横滑→评分网格，视觉密度交替不疲劳。
- **库存三档语义化**（亮点）：>10"仓库有售"（青色安心）/ <10"库存仅剩 N"（红色急迫）/ =0 缺货致电通知——**FOMO 营销心理学**。
- **无算法推荐**，无"猜你喜欢"。

#### 2.3 新蜂（最简）

- **三级分类**：左侧一级 tab（28%）+ 右侧二级标题+三级图文（72%），独立滚动。
- **首页三楼层固定频道**（亮点）：新品上线/热门商品/最新推荐，对应后台 `newGoodses`/`hotGoodses`/`recommendGoodses` 三种运营打标。
- **商品列表**：单列横向卡片，关键词+分类筛选，3 种排序（推荐/新品/价格），上拉无限滚动+下拉刷新。
- **商品详情**：轮播、标题、"免邮费 顺丰快递"固定文案、价格（红色大号）、4 个内容 Tab（概述/参数/安装服务/常见问题，但内容统一为同一富文本，未真正拆分）、底部操作栏。
- **严重缺失**：**完全无 SKU/规格选择**（加车接口只传 `{goodsCount:1, goodsId}`），无库存展示、无评价、无参数独立模块、无相关推荐、无销量/评分。
- **搜索**：仅关键词，无历史/热搜/联想/筛选。

#### 2.4 小结与设计模式提炼

| 维度 | 芋道 | c-shopping | 新蜂 |
| ---- | ---- | ---- | ---- |
| 分类层级 | 多级树 | 三级 | 三级 |
| SKU/规格 | 完整（多规格强制选） | 颜色+尺寸（合并判定有漏洞） | **无** |
| 商品视频 | 无 | 无 | 无 |
| 评价 | 商品级（星级+图+匿名） | **结构化（优点/缺点+5级语义）** | 无 |
| 相关推荐 | DIY 配置 | 相似商品横滑 | 无 |
| 算法推荐 | 无 | 无 | 无 |
| 营销价拼接到列表 | **有（亮点）** | 折扣 badge | 无 |
| 库存语义化 | 下架/无库存灰覆盖 | **三档语义色（亮点）** | 无 |
| 首页结构 | DIY 装修（26 组件） | 7 模块节奏 feed | 三楼层固定 |

**可提炼的设计模式**：
1. **SKU 模型是电商地基**：芋道完整、新蜂缺失——缺失 SKU 等于只能卖单规格商品，这是新蜂不能进生产的根本原因。
2. **结构化评价（c-shopping 优点/缺点 + 5 级语义评级）**：信息密度远超纯文本评价，是内容型电商的差异化护城河。
3. **营销价拼接到列表卡片（芋道）**：用户在列表页就能感知优惠，提升点击率。
4. **库存三档语义色（c-shopping）**：把库存信息转化为购买推动力（FOMO）。
5. **首页运营打标三楼层（新蜂）**：给运营最基础的"内容运营抓手"，简单但适合冷启动。
6. **分类主题色（c-shopping）**：低成本的品牌调性差异化。

### 3. 购物流程

#### 3.1 芋道（价格构成最完整）

- **购物车**：增删改、单选/全选、编辑模式批量删除；服务端返回 **validList（有效）+ invalidList（失效）**，编辑模式合并显示；下架/无库存商品灰色覆盖提示；数量框 min=0（减到 0 自动删除）。
- **结算前校验配送方式冲突**（亮点）：若 A 商品仅支持快递、B 商品仅支持自提，则不允许合并结算。
- **结算页价格构成（7 分项，业界最全）**：① 商品金额 ② 兑换积分（积分商城订单）③ 积分抵扣（可勾选）④ 运费（0 显免运费）⑤ 优惠券（"X 张可用"，弹窗选）⑥ 活动优惠（满减送明细弹窗）⑦ 会员优惠（vipPrice）。
- **业务规则**：拼/秒/积分活动订单**不允许使用优惠券**；地址/配送方式变更触发重新计算。
- **配送方式**：快递发货 + 门店自提双轨，自提需选门店+联系人+手机校验。
- **支付收银台**：`getEnableChannelCodeList` 获取启用渠道，余额支付二次确认，支付倒计时，0 元订单直接跳详情。

#### 3.2 c-shopping

- **购物车本地存储**（Redux + redux-persist + AsyncStorage），**未与服务端同步**（多设备不同步）。
- **巧妙交互**：数量为 1 时减号自动变删除图标（消除 0 数量歧义）。
- **节省金额+百分比双显**（亮点）：强化"占便宜"心理。
- **结算页**：地址卡片、配送说明（"正常发货/有现货"静态文案）、商品预览网格、价格汇总（商品总价/折后总计/节省金额）、支付方式（在线支付/银行卡二选一，**无真实支付集成**）。
- **下单前强制校验地址完整性**（city/province/area/street/postalCode 任一缺失拦截）。
- **无配送方式选择、无发票、无优惠券、无留言**。

#### 3.3 新蜂

- **加车**：每次固定加 1 件（写死 `goodsCount:1`），无规格选择。
- **购物车**：复选框+图+名称+数量步进器（**最小 1 最大 5，硬编码限购**）、单选/全选、左滑删除、底部结算栏实时合计。
- **修改数量防抖**：步进器值与原值相同时不发请求；本地优化：修改后不重新拉接口只改本地（性能优化但牺牲多端一致性）。
- **结算页**：默认地址、商品清单（来自 `/shop-cart/settle` 传 `cartItemIds`）、**仅"商品金额"一项**（无运费/优惠/积分）、固定"全场包邮"。
- **支付**：底部弹窗二选一（支付宝/微信），**纯模拟**（点后直接调 `/paySuccess`）。

#### 3.4 小结与设计模式提炼

| 维度 | 芋道 | c-shopping | 新蜂 |
| ---- | ---- | ---- | ---- |
| 购物车存储 | 服务端（有效/失效分组） | 本地（不同步） | 服务端 |
| 价格分项 | **7 项（最全）** | 3 项（商品/折后/节省） | 1 项（仅商品） |
| 配送方式 | 快递+自提双轨+冲突校验 | 无选择 | 固定包邮 |
| 优惠券介入 | 有（弹窗选券） | 无 | 无 |
| 积分抵扣 | 有（可勾选） | 无 | 无 |
| 真实支付 | 余额+微信+渠道 | 无（模拟） | 无（模拟） |
| 限购 | 活动级限购 | 无 | 单品 5 件硬编码 |

**可提炼的设计模式**：
1. **价格构成可扩展**（芋道 7 分项）：结算页金额应是"可插入优惠项"的管道模型，而非硬编码。
2. **配送方式冲突校验**（芋道）：多配送方式并存时的必要业务规则。
3. **数量=1 减号变删除**（c-shopping）：极简交互范式。
4. **节省金额+百分比双显**（c-shopping）：心理账户强化。
5. **服务端购物车 vs 本地购物车**：服务端同步是多端/多设备前提，本地仅适合 MVP。

### 4. 订单生命周期

#### 4.1 芋道（状态机+动态按钮）

- **状态码**：0 待付款 / 10 待发货 / 20 待收货（已发货）/ 30 已完成 / 40 已关闭；订单项独立 `afterSaleStatus`（0 未申请/10 售后中/20 售后成功）。
- **动态按钮**（后端下发 `buttons` 字段）：cancel/pay/combination（拼团详情）/express/confirm/comment。
- **关键动作**：取消订单（仅待付款）、确认收货（含微信小程序发货信息管理组件 `wx.openBusinessView` 的 `weappOrderConfirm`）、自动收货（后台定时）、删除、订单评价（多商品**逐条评价**，每条含质量评分+服务评分+文字+9 图+**匿名开关**）、再次购买、物流查询（快递公司+单号+轨迹时间线+电话高亮拨号）。
- **自提核销**：自提订单生成核销码，门店端核销。
- **微信小程序订单中心对接**：path 配置、payOrderNo 反查、`wechat_extra_data` 关联。

#### 4.2 c-shopping（极简二元）

- **二元状态**：`delivered` 布尔（完成/未确认）。
- **缺失**：无取消 UI、无确认收货 UI、无退货 UI、无物流跟踪、无按状态筛选/搜索订单。
- **评价入口**：订单卡片未提供"去评价"，只能在商品详情发起。
- **日期 bug**：用 `moment-jalaali`（伊朗贾拉利历），与中文环境不符。

#### 4.3 新蜂（5 阶状态机，亮点）

- **5 阶状态机**（比常见 4 态多一个"待确认"）：0 待付款 / 1 待确认（支付后等后台确认）/ 2 待发货 / 3 已发货 / 4 交易完成 / <0 已取消。
- **状态驱动按钮显隐**（亮点）：详情页按钮完全由 `orderStatus` 决定——
  - 0 待付款：去支付+取消
  - 1 待确认 / 2 待发货：取消
  - 3 已发货：确认收货+取消
  - 4/<0：无按钮
- **二次确认防误操作**：取消订单、确认收货均 dialog 二次确认。
- **缺失**：无删除订单、无评价、无再次购买、无物流查询、无退款/售后、无开票。
- **"待确认"中间态的业务价值**：适配需要人工审单的反欺诈场景（支付后人工确认再发货）。

#### 4.4 litemall 现状（对照）

litemall 订单状态机最细致（11 个状态码 101~402），区分用户取消/系统超时取消/管理员取消、用户申请退款/管理员退款确认、团购超时、系统自动收货等。订单可选操作矩阵按状态明确列出（可取消/可付款/可退款/可删除/可评论/可再买/可收货/可申请售后）。

#### 4.5 小结与设计模式提炼

| 维度 | 芋道 | c-shopping | 新蜂 | litemall |
| ---- | ---- | ---- | ---- | ---- |
| 状态机粒度 | 5 状态+按钮动态下发 | 二元布尔 | 5 阶+状态驱动按钮 | **11 状态（最细）** |
| 自动取消/收货 | 后台定时 | 无 | 推测后台 | 有（定时任务） |
| 物流查询 | 轨迹时间线+电话拨号 | 无 | 无 | 有（快递鸟） |
| 订单评价 | 逐条+服务评分+匿名 | 商品详情发起 | 无 | 订单级评价 |
| 售后入口 | 订单项级 | 无 | 无 | 订单整体级 |
| 微信订单中心 | **深度对接（亮点）** | 无 | 无 | 无 |

**可提炼的设计模式**：
1. **状态驱动按钮显隐**（新蜂+芋道）：后端下发 `buttons` 数组，前端不硬编码业务规则——这是有状态实体详情页的标准范式。
2. **"待确认"中间态**（新蜂）：适合反欺诈审单场景，是 4 态商城的可选增强。
3. **订单项级售后**（芋道）vs 订单整体售后（litemall）：商品级售后是现代电商标配。
4. **微信小程序订单中心+发货信息管理对接**（芋道）：合规要求，litemall 完全缺失。
5. **二次确认防误操作**（新蜂）：不可逆动作的 UX 标配。

### 5. 售后体系

#### 5.1 芋道（最完整）

- **申请类型**：10 仅退款（待发货/待收货/已完成可申请）/ 20 退款退货（订单状态>10 可选，待发货自动移除该选项）。
- **粒度**：**订单项级别**（`orderItemId`），非订单级别。
- **退款金额**：默认取订单项支付金额。
- **申请原因**：后台配置字典（`afterSaleRefundReasons` 仅退款原因 / `afterSaleReturnReasons` 退货原因），弹窗选择。
- **申请描述**（120 字）+ 上传图片（最多 9 张）。
- **状态机**：10 申请已提交 / 20,30 处理中 / 40,50 已完成（退款成功）/ 61,62,63 已拒绝/取消。
- **动态按钮**：cancel 取消申请 / delivery 填写退货。
- **退货物流**：选物流公司+填单号。
- **售后进度时间线**：每步操作时间、操作人、动作描述。
- **售后列表**：5 Tab（全部/申请中/处理中/已完成/已拒绝）。

#### 5.2 c-shopping / 新蜂

- **完全无售后**（c-shopping 仅 SVG 图标"7 天退货"装饰；新蜂无任何售后）。

#### 5.3 litemall 现状

- 售后类型：0 未收货退款 / 1 已收货无需退货退款 / 2 用户退货退款。
- 售后状态：0 可申请 / 1 用户已申请 / 2 管理员审核通过 / 3 管理员退款成功 / 4 管理员拒绝 / 5 用户已取消。
- **限制**：仅订单整体售后，不支持订单商品单个售后；一个订单只能一条售后记录。

#### 5.4 设计模式提炼

1. **订单项级售后**（芋道）是现代电商标配，litemall 的订单整体售后是明显短板。
2. **售后原因字典化 + 多类型分流**（芋道）：仅退款原因与退货原因分离，可后台维护。
3. **售后进度时间线**（芋道）：售后是高焦虑场景，进度可视化是信任建设关键。

### 6. 营销玩法（差异最大的维度）

#### 6.1 芋道（10+ 种玩法，业界最全）

| 玩法 | 业务规则要点 |
| ---- | ---- |
| **优惠券** | 满减/折扣两种；固定日期/领取后 N 天两种有效期；限量（-1 不限）；商品范围全品/指定商品/指定分类；领券中心+商品详情领券+结算页选券+DIY 投放 |
| **满减送** | 多档规则；折扣类型满减/折扣；商品范围全品/指定商品/指定分类；商品详情+结算页+独立专题页 |
| **限时折扣** | SKU 维度，promotionType=4，直降金额+剩余库存+倒计时 |
| **会员折扣** | promotionType=6，订单页 vipPrice 分项 |
| **拼团** | 团长+团员模型，必须达成 userSize 才成功，超时自动失败退款，拼团专属海报，分享支持人数 |
| **秒杀** | 多时间段场次（已结束/进行中/即将开始），总限购+单次限购置最小，库存=min(SKU库存,活动库存)，不走购物车 |
| **积分商城** | 纯积分或积分+现金混合兑换，限购，结算页隐藏优惠券显兑换积分 |
| **分销** | 见第 7 节 |
| **直播** | DIY 组件已就绪（s-live-block/s-live-card），业务待对接 |
| **砍价** | 枚举预留，前台未实现 |
| **组合套餐** | 预留 |

- **营销活动聚合查询**（亮点）：`getActivityListBySpuId(spuId)` 一次性返回某 SPU 当前参与的拼团/秒杀/砍价/积分活动，商品详情页以"活动跳转提示"呈现。
- **结算页一次性聚合所有优惠**：用户无须感知底层复杂度。

#### 6.2 c-shopping（极简）

- **单品折扣**（`discount%`）贯穿商品卡/详情/购物车/订单。
- **首页"折扣商品"专属 feed** + **筛选器"仅限特价"**。
- **购物车节省金额展示**（金额+百分比）。
- **无**：优惠券、秒杀、限时、捆绑、拼团、砍价、referral、积分商城。

#### 6.3 新蜂（几乎空白）

- **三种商品运营标记**（新品/热销/推荐）→ 首页三楼层。
- **支付方式选择**（仅通道，无支付营销）。
- **全场包邮**（固定规则）。
- **首页金刚区 4 个营销概念占位**（9.9 元拼/领劵/省钱/充值缴费），**点击均弹"敬请期待"**。
- **无任何真实促销引擎**。

#### 6.4 litemall 现状

- **优惠券**（较完整）：3 种类型（通用领取/注册赠/兑换码）、3 种商品限制（全品/类目/指定商品）、2 种有效期（固定天数/起止时间）、限领数量、库存总量。
- **团购**（完整）：团购规则（人数门槛/优惠金额/过期时间）→ 团购活动（开团/参团/成功/失败）→ 团购订单。
- **评价**（双类型：商品评论+专题评论）。
- **专题**（富文本+关联商品）。
- **首页 Banner**（`litemall_ad` position=1）。

#### 6.5 设计模式提炼

1. **营销矩阵分层**：基础促销（满减/限时折扣/会员价）→ 流量玩法（秒杀/拼团/砍价）→ 留存玩法（积分商城/签到）→ 裂变玩法（分销）→ 内容玩法（直播）。芋道覆盖全部分层，litemall 仅基础促销+流量玩法（团购）。
2. **营销活动聚合查询**（芋道）：商品页统一展示所有可参与活动，避免用户在不同入口反复跳转。
3. **营销玩法组件化**（芋道 DIY）：拼团/秒杀/积分/优惠券均可作为装修块拖入页面，运营自主性极高。
4. **满减引擎缺失是 litemall 的明显短板**：litemall 订单总价仅"优惠券减免+团购减免"，无满减引擎，而满减是最基础最高频的促销。

### 7. 分销体系

#### 7.1 仅芋道实现（二级分销完整闭环）

- **分销商权限**：后台开通，进入分销中心校验 `brokerageEnabled`。
- **分销中心**：可提现金额、冻结金额、累计佣金、佣金流水。
- **佣金规则**：商品维度配置 min/max 佣金区间；佣金冻结期（`brokerageFrozenDays`）；最低提现金额（`brokerageWithdrawMinPrice`）。
- **二级分销**：一级/二级 Tab 切换，推广人数总览，团队成员列表（头像/昵称/加入时间/推广人数/订单数/佣金），三种排序（团队/金额/订单），昵称搜索。
- **5 种提现方式**：钱包余额 / 银行卡转账 / 微信账户（收款码）/ 支付宝账户（收款码）/ 微信零钱（企业付款到零钱通道）。
- **排行榜**：推广人排行（按用户量）/ 佣金排行（按佣金，时间段筛选）。
- **推广海报**：用户海报/商品海报/拼团海报，canvas 端侧生成。
- **推广人绑定**：邀请链接携带 `spm` 参数，首页 onLoad 解析→解密→绑定上下级，登录后自动 PUT 绑定；小程序二维码 scene 也支持携带绑定参数（**全自动，无需手动输邀请码**）。

#### 7.2 c-shopping / 新蜂 / litemall

- **全部无分销**。

#### 7.3 设计模式提炼

1. **spm 全自动绑定**（芋道）：分享即绑定，无需用户手动输邀请码，是分销裂变的关键体验。
2. **佣金冻结期 + 最低提现**：风控标配，防止刷单套现。
3. **多渠道提现**：适配不同分销商群体（个人/个体户/企业）。
4. **分销是 litemall 完全空白的高价值玩法**，但实现复杂度高（佣金计算、冻结、提现、风控、税务），需独立规划。

### 9. 客户服务

#### 9.1 芋道（业务深度最高）

- **意见反馈**：前端未独立提供（预留），通常通过客服会话或后台工单承接。

#### 9.2 c-shopping

- **无在线客服、无 IM、无客服入口**。
- Feed 头部"通知铃铛"→ `/notice` 页**仅占位文本"Notice Screen"未实现**。

#### 9.3 新蜂

- **无客服**：商品详情页底部"客服"图标**仅装饰无功能**。
- **无消息中心、无帮助中心、无意见反馈**。
- 关于我们静态页（项目简介+开源地址）。

#### 9.4 litemall 现状

- **客服**：微信原生 `<contact-button>` 转发到微信客服 + `WxMsgController` 消息转发，**无自建 IM、无客服工作台、无会话记录**。

#### 9.5 设计模式提炼

1. **会话内发送商品/订单卡片**（芋道）大幅提升客服效率（用户不用描述，客服直接看）。
2. **客服已读回执**（芋道）是高焦虑场景的信任建设。
3. **自建 IM vs 微信客服转发**：自建 IM（芋道）可控可存档可分析，微信转发（litemall）零成本但不可控。
4. **服务承诺图标套装**（c-shopping）是信任建立的视觉语汇，低成本。

### 11. 设置与其他

#### 11.1 芋道

- **微信小程序订阅消息**三类：订单发货 / 拼团结果 / 充值成功，业务动作触发时拉起订阅授权。

- **我的收藏**：分页+收藏/取消+实时状态。
- **浏览历史/足迹**：分页+单条删除+清空全部。
- **设置**：应用信息、版本（检查更新）、本地缓存大小、关于我们、用户协议、隐私协议、**注销账号（仅 iOS App 显，合规过审）**、退出登录。
- **多租户支持（SaaS）**：H5 按 URL 参数 tenantId>域名 host 反查，小程序按 appId 反查；切换租户清 token 重定向。
- **全局分享**：统一生成分享信息，分享场景（首页/普通商品/拼团商品/秒杀商品/参与拼团/积分商品），推广员分享携带 spm 自动绑定上下级。
- **错误页与 webview**：通用错误页+内嵌 webview 容器。

#### 11.2 c-shopping

- **地址簿**：单地址设计（非多地址簿），三级中国行政区划联动+街道+邮编，无默认地址/地址标签/地图选点。
- **多语言/多币种/主题**：无（单一中文/人民币/浅色主题）。
- **推送通知/消息中心**：无。
- **设置**：无独立设置页（无清除缓存/版本/协议/注销）。
- **客户端浏览足迹**（最多 15 条去重自动滚动）。
- **状态体验体系**（亮点）：完整空态/错误态/骨架态体系，每种独立插画+统一错误重试，分页>1 时不打断显示。
- **表单校验体系**：Yup schema 集中管理。

#### 11.3 新蜂

- **设置页实为"账号管理"**（昵称/签名/密码/退出）。
- **关于我们**：静态页。
- **无版本更新/清除缓存/多语言/主题/隐私协议/注销**。

#### 11.4 litemall 现状

- **关于我们**：商城名/地址/电话/QQ/经纬度（地图定位）。
- **地址/收藏/足迹/搜索历史**：完整。
- **站内信**：用户端无。
- **多租户**：单店模型。

#### 11.5 设计模式提炼

1. **iOS 注销账号合规**（芋道）：App Store 审核要求，移动 App 必备。
2. **状态体验完备性**（c-shopping）：空/错/骨架态独立插画+统一重试，直接影响转化率，是开源电商少见的细致。
3. **微信订阅消息**（芋道）：业务事件触发订阅，是小程序消息触达的主流方式。litemall 仅依赖微信模板消息。

## 第三部分：为 litemall 提出的功能增强建议

> 本部分是分析报告的落地输出，按「移动商城功能增强」与「后台设计完善」两条线组织，每条建议标注：来源借鉴（芋道/c-shopping/新蜂）、优先级（P0/P1/P2）、与 litemall 现状的差距、最小可行实现要点。
>
> 优先级口径：P0 = 主链路补全 / 合规要求 / 高 ROI；P1 = 运营核心能力；P2 = 体验增强 / 差异化。

### 3.1 移动商城功能增强建议

#### M-P0-1 会员等级体系（填实伪功能）

- **现状差距**：litemall `litemall_user.user_level` 字段存在（0 普通/1 VIP/2 高级 VIP）但**无任何配套规则、权益、升级逻辑**，形同摆设。
- **借鉴**：芋道会员等级 + 会员价（promotionType=6）。
- **最小实现**：
  - 等级规则（升级条件：累计消费金额/订单数/成长值）、等级权益（专属价/专属券/生日礼包/专享客服）、降级机制（周期内未达标降级）。
  - 订单结算页显示"会员优惠"分项（`vipPrice`）。
  - 个人中心展示当前等级+下一级进度。

#### M-P0-2 积分体系（补全伪功能字段）

- **现状差距**：`litemall_order.integral_price` 字段存在但**无积分账户与流水表**——伪功能。
- **借鉴**：芋道积分（账户+流水+抵扣+兑换）。
- **最小实现**：
  - 积分账户表（用户余额）+ 积分流水表（来源：签到/购物/评价/分享；用途：下单抵扣/积分商城兑换）。
  - 积分获取规则配置（后台）、积分抵扣开关（结算页可勾选）。
  - "我的积分"页（余额+收支 Tab+时间段筛选+流水）。

#### M-P0-3 签到与任务中心（日活抓手）

- **现状差距**：无。
- **借鉴**：芋道签到（可配连续奖励规则）。
- **最小实现**：
  - 后台配置签到规则（第 N 天奖励 X 积分）、连续/累计天数、已签置灰、签到结果弹窗。
  - （可选 P2）新手任务/成长任务扩展为任务中心。

#### M-P0-4 满减/满赠促销引擎（最基础最高频）

- **现状差距**：litemall 订单总价仅"优惠券减免+团购减免"，**无满减引擎**——满减是最基础最高频的促销。
- **借鉴**：芋道满减送（多档规则、满减/折扣、商品范围全品/指定商品/指定分类）。
- **最小实现**：
  - 满减活动表（折扣类型、多档规则、商品范围、起止时间）。
  - 商品详情页展示满减规则（最多 3 条）、结算页"活动优惠"分项+明细弹窗。
  - 独立满减活动专题页（可选 P2）。

#### M-P0-5 限时折扣/会员价（SKU 维度促销）

- **现状差距**：无。
- **借鉴**：芋道限时折扣（promotionType=4，直降+剩余库存+倒计时）、会员折扣。
- **最小实现**：商品级限时价 + 详情页促销横幅（促销价+直降金额+剩余库存+倒计时）。

#### M-P0-6 订单项级售后（现代电商标配）

- **现状差距**：litemall 仅订单整体售后，**不支持订单商品单个售后**，一个订单只能一条售后记录。
- **借鉴**：芋道订单项级售后（`orderItemId`）+ 售后进度时间线。
- **最小实现**：
  - 售后单关联订单项（非订单整体），一个订单可多条售后。
  - 申请类型分流（仅退款/退款退货，按订单状态自动可选）。
  - 售后原因字典化（后台维护仅退款原因/退货原因）。
  - 售后进度时间线（每步操作时间/操作人/动作）。

#### M-P0-7 用户站内信/消息中心

- **现状差距**：litemall `litemall_notice`/`notice_admin` 只面向管理员，**用户端无站内信**。
- **借鉴**：芋道微信订阅消息（订单发货/拼团/充值）+ c-shopping 通知中心（虽占位）。
- **最小实现**：
  - 用户站内信表（订单消息/营销消息/系统消息分类）。
  - 个人中心消息入口+未读徽章+消息列表。
  - （可选）微信小程序订阅消息对接（业务事件触发订阅授权）。

#### M-P0-8 微信小程序订单中心 + 发货信息管理对接（合规）

- **现状差距**：litemall 完全缺失。微信小程序电商监管要求订单中心 path 配置与发货信息管理。
- **借鉴**：芋道 `wx.openBusinessView` 的 `weappOrderConfirm` + `wechat_extra_data` 关联。
- **最小实现**：小程序订单中心 path 配置、payOrderNo 反查、确认收货调起官方组件。

#### M-P1-9 秒杀活动（流量玩法）

- **现状差距**：无。
- **借鉴**：芋道秒杀（多时间段场次、总限购+单次限购、库存=min(SKU库存,活动库存)、不走购物车）。
- **最小实现**：秒杀场次配置+活动列表（已结束/进行中/即将开始+倒计时+抢购进度条）+商品详情秒杀价+限购。

#### M-P1-10 拼团（流量玩法，区别于团购）

- **现状差距**：litemall 仅有"团购 groupon"（规则：人数门槛+优惠金额），**无拼团**（团长+团员+成团模型）。
- **借鉴**：芋道拼团（团长+团员模型、userSize 达成才成功、超时自动失败退款、拼团专属海报）。
- **最小实现**：拼团活动（团长开团/团员参团/成团/失败退款）+拼团详情（团长+团员头像+空位+邀请好友）+我的拼团。

#### M-P1-11 优惠券体系增强（litemall 已较完整，补 DIY 投放）

- **现状差距**：litemall 优惠券较完整（3 类型/3 商品范围/2 有效期），但**无领券中心入口聚合 + 无 DIY 投放**。
- **借鉴**：芋道领券中心+商品详情领券+DIY 投放。
- **最小实现**：领券中心页（聚合所有可领券）+商品详情可领券预览+领券入口。

#### M-P1-12 首页运营打标三楼层（DIY 轻量替代）

- **现状差距**：litemall 首页固定 9 块结构，**完全不支持 DIY**。
- **借鉴**：新蜂三楼层（新品/热销/推荐运营打标）作为 DIY 的轻量替代。
- **最小实现**：商品三标记字段（newFlag/hotFlag/recommendFlag 独立于排序权重）+首页三楼层展示。
- **注**：完整 DIY 装修（芋道 26 组件）实现复杂度极高，列为 P2 或独立大特性。

#### M-P1-13 商品评价增强（结构化 + 标签统计）

- **现状差距**：litemall 评价为双类型（商品/专题）+星级+多图+管理员回复，**无结构化、无标签统计、无评价筛选**。
- **借鉴**：c-shopping **优点/缺点结构化评价 + 5 级语义评级**（最大亮点）。
- **最小实现**：评价表单增加优点/缺点动态列表+5 级语义文案；评价展示增加好评率/标签云/有图筛选。

#### M-P1-14 商品视频 + 商品问答

- **现状差距**：litemall 商品仅图片 gallery，**无视频、无问答**。
- **借鉴**：三项目均无视频，但视频是现代电商标配。
- **最小实现**：商品视频字段（详情页顶部视频播放）+商品问答（买家问卖家答+公开展示）。

#### M-P1-15 钱包余额 + 充值赠送

- **现状差距**：litemall 无钱包、无余额。
- **借鉴**：芋道钱包（余额+充值套餐+充值赠送+流水+收支汇总）。
- **最小实现**：钱包账户表+充值套餐配置（固定金额+赠送）+流水+充值走支付+充值记录页。

#### M-P1-16 多支付通道（支付宝 + 余额）

- **现状差距**：litemall 仅微信支付。
- **借鉴**：芋道多渠道（余额+微信+其他）。
- **最小实现**：支付宝 H5/小程序支付接入+余额支付+收银台渠道选择。

#### M-P1-17 配送方式扩展（自提 + 同城配送）

- **现状差距**：litemall 仅快递。
- **借鉴**：芋道快递+自提双轨+冲突校验+自提核销码。
- **最小实现**：自提门店表+自提订单核销码+结算页配送方式切换+配送方式冲突校验。

#### M-P2-18 库存语义化 + 营销价拼接到列表

- **现状差距**：litemall 无库存语义、列表无营销价。
- **借鉴**：c-shopping 库存三档语义色 + 芋道营销价拼接到列表卡片。
- **最小实现**：列表/详情库存三档展示 + 列表卡片拼接当前生效促销价。

#### M-P2-19 状态体验完备性（空/错/骨架态）

- **现状差距**：litemall 状态体验基础。
- **借鉴**：c-shopping 完整空态/错误态/骨架态体系（每种独立插画+统一错误重试+分页>1 不打断）。
- **最小实现**：统一 ShowWrapper 模式+各场景独立插画+骨架屏+错误重试。

#### M-P2-20 分销体系（独立大特性）

- **现状差距**：litemall 完全空白。
- **借鉴**：芋道二级分销完整闭环（spm 全自动绑定、佣金冻结期、5 种提现、排行榜）。
- **注**：实现复杂度高（佣金计算/冻结/提现/风控/税务），建议独立规划，列为 P2 或独立阶段。

#### M-P2-21 DIY 可视化装修（独立大特性）

- **现状差距**：litemall 完全不支持。
- **借鉴**：芋道 26 组件 DIY 装修（模板+页面+组件体系）。
- **注**：实现复杂度极高（前后端装修编辑器+组件体系+模板机制），建议独立规划，列为 P2 或独立阶段。M-P1-12 三楼层可作为其过渡方案。

#### M-P2-22 在线客服 IM（自建）

- **现状差距**：litemall 仅微信原生 `<contact-button>` 转发，无自建 IM。
- **借鉴**：芋道 WebSocket IM（商品/订单卡片+已读回执）。
- **注**：自建 IM 实现复杂度高，若预算有限可继续用微信客服转发。

### 3.2 后台设计完善建议

#### A-P0-1 Dashboard 重做（替换 4 数字板）

- **现状差距**：litemall Dashboard 极简（仅 4 个累计总数：用户/商品/货品/订单）。
- **借鉴**：通用电商后台标准。
- **最小实现**：实时核心指标卡（今日 GMV/UV/转化率/客单价/退货率）+销售趋势图（时/天/周/月）+同环比+实时订单流+待办事项（待发货/待退款/售后待审核/库存预警）。

#### A-P0-2 数据报表体系扩展（三大主题域深度）

- **现状差距**：litemall 仅 3 张按天简单表（用户/订单/商品）。
- **借鉴**：通用电商报表标准。
- **最小实现**：
  - 销售漏斗（浏览→加购→下单→支付→复购）。
  - 用户分析（留存次留/7 留/30 留、RFM 分层、复购率、生命周期）。
  - 商品分析（销量排行、加购排行、滞销品、动销率、毛利）。
  - 营销分析（优惠券核销率、活动 ROI）。
  - 订单分析（客单价分布、支付方式占比、退货原因）。
  - 自定义报表+数据导出。

#### A-P0-3 用户运营工作台（替换 6 个只读列表）

- **现状差距**：litemall 会员管理 6 个菜单（会员/地址/收藏/足迹/搜索历史/反馈）**全是只读列表**，无任何运营操作。
- **借鉴**：通用电商用户运营标准。
- **最小实现**：用户标签+用户分群+用户详情页（订单/足迹/积分/客服记录聚合）+用户封禁/解封+调整等级+手工发券+手工加积分+用户画像+黑名单。

#### A-P0-4 订单运营工作台

- **现状差距**：litemall 订单管理仅列表/详情/发货/退款/回复/删除。
- **借鉴**：通用电商订单运营标准。
- **最小实现**：批量发货（Excel 导入单号）+批量打印快递单/发货单+订单改价/改运费+订单标记/备注/打标+订单异常监控（超期未发货/未支付）+退款审核工作台+发货工作台（合单/拆单/改地址）+发票管理+**订单号搜索**（文档自承认缺失）。

#### A-P0-5 营销活动管理（满减/秒杀/拼团后台）

- **现状差距**：litemall 推广管理仅广告/专题/优惠券/团购规则/团购活动。
- **借鉴**：芋道营销矩阵后台。
- **最小实现**：满减活动配置（多档/商品范围）+秒杀场次配置+拼团活动配置+营销活动日历（排期+冲突检测）+活动效果分析（参与人数/GMV/ROI）+优惠券使用统计（领取率/核销率/拉动 GMV）。

#### A-P1-6 商品运营增强

- **现状差距**：litemall 商品管理 CRUD+上下架+评论删除。
- **借鉴**：通用电商商品运营标准。
- **最小实现**：批量导入/导出商品（Excel/CSV）+批量改价/改库存/上下架+库存预警（低库存提醒+安全库存）+商品组合/套装+**评论审核工作台**（当前只能删除）+评价回复工作台。

#### A-P1-7 财务模块

- **现状差距**：litemall 无财务模块。
- **借鉴**：通用电商财务标准。
- **最小实现**：交易对账+退款对账+利润报表（成本价+毛利）+提现审核（分销场景）+结算报表。

#### A-P1-8 内容运营增强

- **现状差距**：litemall 专题直接发布无审核，无素材库，无 SEO。
- **借鉴**：通用电商内容运营标准。
- **最小实现**：专题发布工作流（审核）+内容素材库（图片/视频统一管理）+SEO 配置（标题/描述/关键词）。

#### A-P1-9 权限与多店

- **现状差距**：litemall 文档明确"只完成操作权限，**数据权限未完成**"，单店模型。
- **借鉴**：通用电商多店标准。
- **最小实现**：数据权限（门店隔离）+多门店/多店铺（门店表+店员-门店关系）+多角色协作工作流（审核流）。

#### A-P2-10 系统管理增强

- **现状差距**：litemall **无字典管理、无独立登录日志**（与操作日志混在一起）。

## 第四部分：关键风险与权衡提示

### 4.1 借鉴时的反向约束（不可盲目照搬）

1. **芋道是 SaaS 多租户**，litemall 是单店模型。若 litemall 不做多租户，芋道的租户识别/租户切换/跨租户数据隔离等设计**不可直接照搬**，只取业务玩法部分。

2. **新蜂支付是纯模拟**，litemall 已有真实微信支付。新蜂的"待确认"中间态适合反欺诈审单场景，但若 litemall 业务不需要人工审单，**强加中间态反而增加运营负担**——需按业务确认。

3. **c-shopping 购物车是本地存储不同步**，这是 MVP 取舍，litemall 已是服务端购物车，**不可退化**。c-shopping 的 SKU 合并判定漏洞（颜色+尺寸只按一维）**必须避免**。

4. **c-shopping 日期用伊朗贾拉利历**（`moment-jalaali`）是源项目残留 bug，**不可照搬**。

5. **新蜂首页金刚区 100% 装饰**（点击全弹"敬请期待"），litemall 若引入金刚区**必须真正实现或下线**，装饰性金刚区损害体验。

6. **新蜂无 SKU 模型**是教学简化，litemall 已有完整 SKU（`litemall_goods_product`），**不可退化**。

### 4.2 大特性的独立规划必要性

以下建议实现复杂度高，**不应混入常规迭代，需独立规划**：

- **DIY 可视化装修**（M-P2-21）：前后端装修编辑器+26 组件体系+模板机制，工作量等同独立产品。
- **分销体系**（M-P2-20）：佣金计算/冻结/提现/风控/税务/排行榜，涉及资金流。
- **自建 IM 客服**（M-P2-22）：WebSocket+会话存档+客服工作台+商品/订单卡片，涉及实时通信。
- **多租户 SaaS**：若未来需要，是架构级改造。

### 4.3 合规与平台约束

- **微信小程序订单中心 + 发货信息管理**（M-P0-8）是监管合规要求，**不是可选项**——若 litemall 小程序上架电商类目必须对接。
- **iOS 注销账号**（借鉴芋道）是 App Store 审核要求，若做 iOS App 必备。
- **7 天无理由退货**是电商法规定要求，litemall 已有售后但**仅订单整体级**，订单项级售后（M-P0-6）是合规升级方向。

### 4.4 与 nop-app-mall 现有设计的关系

> 注：本报告分析对象是 litemall（增强目标），但当前仓库是 nop-app-mall（litemall 的 Nop 平台重实现）。nop-app-mall 的 `docs/design/` 已有 owner doc 覆盖大部分 P0 建议（见 `docs/design/marketing-and-promotions.md`、`order-and-cart.md`、`user-and-address.md` 等）。

- 本报告的 litemall 增强建议**不直接修改 nop-app-mall 的设计**，而是作为 litemall 增强和后续功能设计的参考输入。
- 若后续要在 nop-app-mall 落地其中某些建议，应遵循 nop-app-mall 的 owner doc 体系，先更新 `docs/requirements/` 或对应 `docs/design/` owner doc，而非直接照搬本报告。

## 第五部分：结论

### 5.1 三项目的差异化价值定性

- **芋道**：功能广度第一梯队，是"功能参考最大金矿"。其营销矩阵、价格构成管道、DIY 装修、分销闭环、微信生态深度对接、SaaS 多租户从端到端贯通，几乎覆盖现代电商所有玩法。**适合作为功能清单的权威参考**。
- **c-shopping**：功能广度克制，但 UI/UX 设计深度第一梯队。其结构化评价、首页 feed 节奏、库存心理学、状态体验完备性，是"深度优于广度"的设计模式典范。**适合作为体验设计的差异化灵感**。
- **新蜂**：功能广度最小，但订单状态机设计教学价值最高。其 5 阶状态机、状态驱动按钮、运营打标三楼层，清晰展示电商骨架的最小可运行实现。**适合作为主流程骨架的教学参考**。

### 5.2 litemall 增强的核心结论

litemall 的功能基线本质是"**单店、单支付通道、无分销、无 IM、无装修的小型 B2C 自营商城参考实现**"——交易主链路完整（11 态订单状态机最细），但**深度运营能力薄、营销玩法少、报表基础、会员/积分/钱包伪功能字段待填实**。

增强的核心方向：

1. **移动商城**：先填实伪功能（会员等级/积分）+ 补基础促销（满减/限时/会员价）+ 升级售后到订单项级 + 补合规（微信订单中心）+ 补消息触达（站内信）。这是 P0。
2. **后台设计**：Dashboard 重做 + 报表体系扩展 + 用户/订单运营工作台（从只读变可操作）+ 营销活动管理。这是 P0。
3. **大特性独立规划**：DIY 装修、分销、自建 IM、多租户——不混入常规迭代。

### 5.3 后续行动建议

1. 将本报告第三部分的建议清单转化为 `docs/backlog/` 的候选工作项（按 nop-app-mall 的工作流，若目标是 litemall 则转化为 litemall 的 backlog）。
2. 每个建议落地前，先在 `docs/requirements/` 或对应 owner doc 沉淀业务语义，再进入实现。
3. 大特性（DIY 装修/分销/自建 IM）单独立项，先做需求综合与架构评估。
4. 合规项（微信订单中心/订单项级售后）优先排期，避免平台下架风险。

## 附录 B：术语表

- **SPM**（芋道）：Share + Promoter Mark，推广人标识参数，用于分享链接自动绑定分销上下级。
- **DIY 装修**：可视化拖拽编辑商城页面（首页/个人中心/自定义页）的运营能力。
- **订单项级售后**：以订单内单个商品为单位申请售后（区别于订单整体售后）。
- **待确认中间态**（新蜂）：订单支付后、发货前的后台人工审单状态，用于反欺诈。
- **FOMO**：Fear Of Missing Out，错失恐惧，库存语义化、限时倒计时等营销心理学的理论基础。
- **RFM**：Recency-Frequency-Monetary，用户价值分层模型（最近购买时间/购买频率/消费金额）。
