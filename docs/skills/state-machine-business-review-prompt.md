# 状态机业务审查提示词

## 使用方式

将以下提示词提供给审查者（子 agent 或人工），附上状态机的完整设计文档（状态列表、迁移矩阵、触发动作、角色权限、业务说明），按提示词逐项审查。

---

## 提示词正文

你是一名资深业务分析师和技术架构师。下面是一个状态机设计文档，请按以下审查维度逐项审查，指出每个维度中的**业务逻辑错误、遗漏、矛盾和不合理之处**。审查时不要只停留在表面，要代入实际业务流程推演每个状态迁移场景。

### 一、状态定义审查

#### 1.1 状态命名
- 每个状态名称是否清晰表达该阶段的业务含义？
- 名称是否容易与外部对象状态（第三方系统返回的状态）混淆？
- 是否存在命名上看起来是"动作"而非"状态"的节点？（状态是"等待中"而非"去做某事"）

#### 1.2 状态完备性
- 从业务起点到终点的全链路是否都有对应的状态？
- 是否存在业务上必须存在、但状态机中没有定义的"空白期"？
- 所有异常退出路径（取消、退回、拒绝、超时关闭）是否都有对应的状态覆盖？

#### 1.3 状态冗余
- 是否存在两个状态在业务语义上其实表示同一件事？（如"待审核"和"审核中"实际上无法区分）
- 是否存在仅凭业务字段就能区分、不需要单独建状态的情况？
- 是否存在为"动作"而不是"等待点"建立的状态？（如果某状态不需要人工等待，是否应该降级为节点内动作？）

### 二、状态迁移审查

#### 2.1 迁移完整性
- 对于每个状态，是否明确列出了所有可以到达该状态的前置状态？
- 对于每个状态，是否明确列出了所有可以从该状态迁移出去的目标状态？
- 是否存在某个业务操作会导致状态变化，但迁移矩阵中没有定义对应的迁移？

#### 2.2 业务语义正确性
- 逐条检查每条迁移路径，问自己：**"在真实业务中，从A到B这条路径是否合理？"**
- 是否存在跳跃性迁移？例如从"初审"直接跳到"终审"跳过了"复审"——这是否符合业务规则？
- 是否存在不该发生的反向迁移？例如退回后应该进入"草稿"还是"待补充"？
- 是否存在不该发生的正向迁移？例如未完成资料审核就进入"处理中"？

#### 2.3 迁移条件
- 如果某个迁移需要满足特定业务条件（如金额阈值、资料完整性、外部系统结果），这些条件是否在设计中体现了？
- 是否存在"条件A下走路径1、条件B下走路径2"的迁移分叉，但在状态机设计中没有区分？
- 对于需要外部系统触发的迁移（轮询、回调），是否明确了触发机制？

### 三、终态与回收审查

#### 3.1 终态合理性
- 列出所有终态（没有出边的状态），确认这些确实是业务上允许结束的状态。
- 是否存在不该是终态的状态被错误地设为了终态？（例如"对账处理中"不应该是终态）
- 终态之间的互斥性：是否存在同时满足两个终态条件的可能？

#### 3.2 回收流
- 从终态是否可以重新激活回到流程中？（例如争议关闭后发现新问题能否重开）
- 如果存在回收/重开路径，是否在状态机中明确画出？
- 归档案件和活跃案件的数据是否在同一张表？

### 四、异常路径审查

#### 4.1 异常场景覆盖
以下是常见的异常场景，检查状态机是否覆盖：

- **超时**：节点超过 SLA 时限未处理，状态应如何变化？
- **拒绝/退回**：审核不通过时，是退回上一步还是退回起点？
- **取消/撤销**：流程中途是否可以取消？取消后能否恢复？
- **外部系统异常**：外部 API 调用失败时，状态是否回滚或进入异常等待？
- **重复操作**：同一操作被重复执行时，状态机如何处理？
- **并发操作**：多人同时操作同一案件时，状态机是否有防重入或乐观锁机制？

#### 4.2 幂等性
- 相同操作被重复触发（如轮询服务重启后重复处理），状态迁移是否幂等？
- 如果当前状态已经是目标状态，再次触发迁移是否应该报错还是静默跳过？

### 五、可达性分析

#### 5.1 所有状态可达
- 从起始状态（S0）出发，是否存在不可达的状态？（例如某个状态的入边为空，或入边的前置状态本身不可达）
- 从起始状态出发，能否通过某种路径到达每个应达的状态？

#### 5.2 死锁/死循环
- 是否存在无法到达任何终态的路径？（例如 A→B→A→B... 无限循环，且没有终止条件）
- 是否存在状态迁移环路（A→B→C→A），这是否是业务上合理的循环？
- 如果是合理的业务循环，是否有退出条件？（例如补充资料最多退回3次）

### 六、角色与操作权限审查

#### 6.1 操作绑定
- 每个状态迁移（操作）是否绑定了执行角色？
- 是否存在"任何角色都能执行"的危险操作？（如确认结案）
- 角色的职责边界是否清晰？是否与组织架构中的人员分工一致？

#### 6.2 角色与状态一致性
- 角色名称和状态名称是否来自同一套业务语言？（如"风控岗"对应"风控审核中"）
- 是否存在一个状态可以由多个角色操作的情况？此时如何处理冲突？

### 七、外部依赖审查

#### 7.1 外部系统状态与内部状态的关系
- 外部系统返回的状态（如卡组织返回"ACCEPTED"）是否直接映射为内部状态，还是需要封装？
- 外部状态变化是否通过轮询、回调还是人工触发进入内部状态机？
- 外部系统超时/不可用时，内部状态是否有兜底策略？

#### 7.2 异步处理
- 异步通知（Webhook、MQ）到达时，状态迁移是否考虑了消息顺序和去重？
- 轮询任务多次执行时，状态迁移是否幂等？

### 八、待办/TODO 策略审查

#### 8.1 每个状态的待办策略
- 每个非终态是否明确产生了待办任务？
- 待办类型（池任务/已分配/监控/确认）是否与业务场景匹配？
  - "需要人工决定"的节点应该是"已分配"或"池任务"
  - "只需系统等待"的节点应该是"监控类"
  - "系统准备好、等人确认"的节点应该是"确认类"
- 是否存在"有人等着做"但没有产生待办的状态？（会导致案件"沉没"）

#### 8.2 自动流转
- 轮询/定时任务自动触发的迁移，是否产生了待办任务？
- 自动迁移到达新状态后，新状态是否产生了正确的待办类型？

### 九、业务场景反推（最重要的审查方式）

选择以下2-3个典型业务场景，沿着状态机完整走一遍，看是否走得通：

1. **正常流程**：走最理想的路径（资料完整→审核通过→处理完成→结案），检查每步状态迁移
2. **拒绝/退回流程**：在审核节点被退回，检查状态是否回退、重新提交后是否走回正常路径
3. **异常终止流程**：在某个节点取消案件，检查终态是否正确
4. **外部触发流程**：外部系统状态变化触发状态迁移，检查是否符合业务预期
5. **超时流程**：某个节点超时未处理，检查状态是否自动推进或告警

### 十、与设计文档的一致性审查

- 状态机中的每个状态是否在设计文档中都有对应的页面、接口、权限和业务说明？
- 设计文档中的每个业务操作是否在状态机中找到了对应的迁移？
- 是否存在"设计文档写了但状态机没实现"或"状态机定义了但设计文档没描述"的不一致？

## 常见状态机错误模式清单

审查时对照以下已知的反模式（anti-patterns）：

| 反模式 | 错误表现 | 正确做法 |
|--------|---------|---------|
| **动作为状态** | 把"创建退单"作为状态 | 创建退单是动作，等待退单结果是状态 |
| **状态遗漏** | 审核未通过退回后，案件"悬空"无状态 | 退回应有明确状态（如待补充/草稿） |
| **非法迁移** | 从"待审核"跳到"已结案" | 必须经过中间状态 |
| **单向循环** | A→B→C→A 的三方循环无退出条件 | 循环必须有终止条件 |
| **角色漂移** | 操作者角色与状态不匹配（初审岗做结案操作） | 每个操作绑定正确角色 |
| **沉默状态** | 状态存在但不产生待办，案件无人处理 | 非终态必须产生明确待办 |
| **条件即状态** | 为每个条件分支建状态 | 用业务字段区分分支，状态只表达"等待什么" |
| **外部状态裸映射** | 直接把第三方API返回值当状态 | 封装为内部业务状态后再映射 |
| **终态不是终态** | CLOSED 状态还有出边 | 终态定义必须与业务终局一致 |

## 审查输出格式

按以下格式输出审查结果：

```
### [严重度] 维度: 问题描述
- **位置**: 具体状态/迁移
- **问题**: 为什么是错的
- **建议**: 如何修改
- **参考**: 相关业务逻辑或文档依据
```

严重度分级：
- **P0（阻断）**：会导致业务无法走通或数据错误
- **P1（严重）**：会导致错误的状态变化或案件异常
- **P2（一般）**：不符合最佳实践但现有路径能走通
- **P3（建议）**：优化性建议
